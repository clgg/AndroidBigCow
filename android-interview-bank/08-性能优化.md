# 性能优化

## Q: Android 冷启动过程有哪些阶段？如何优化？

标签：高频 / 性能 / 场景题

复习状态：未掌握

考察点：

- 进程创建。
- Application 和首屏。
- 启动指标。

答案要点：

- 冷启动包括 Zygote fork 进程、创建 Application、启动 Activity、加载布局、首帧绘制。
- 优化方向包括减少 Application 初始化、延迟非关键任务、异步预加载、优化首屏布局和图片。
- 应用启动要以可观测指标为准，例如 TTID、TTFD、首帧时间和业务可交互时间。
- Android 可结合 Baseline Profiles、启动跟踪和 Perfetto 分析。

深挖追问：

- ContentProvider 为什么会拖慢启动？
- 冷启动和热启动怎么区分？
- 启动优化如何避免异步任务挤占主线程？

常见误区：

- 把所有初始化都丢到子线程但不控制依赖。
- 只看日志时间，不看系统级 trace。

## Q: 内存泄漏如何定位和治理？

标签：高频 / 内存 / 场景题

复习状态：未掌握

考察点：

- 引用链。
- 生命周期。
- 工具使用。

答案要点：

- 内存泄漏是对象已经无业务价值但仍被强引用持有，导致无法回收。
- 常见来源包括静态引用、Handler 延迟消息、监听器未注销、线程任务、单例持有 Context。
- 可用 LeakCanary、Heap Dump、Android Studio Profiler 分析引用链。
- 治理重点是缩短引用生命周期、使用 Application Context、及时反注册和清理任务。

深挖追问：

- 如何判断泄漏是否严重？
- Bitmap 内存如何排查？
- native 内存泄漏怎么查？

常见误区：

- 看到内存上涨就认为一定泄漏。
- 只加 WeakReference 不处理真正生命周期。

## Q: 卡顿问题如何系统排查？

标签：高频 / 性能 / 场景题

复习状态：未掌握

考察点：

- 16.6ms 帧预算。
- 主线程耗时。
- Trace 工具。

答案要点：

- 卡顿通常是某一帧或连续多帧未能按时完成输入、动画、布局、绘制、同步和渲染。
- 常见原因包括主线程 IO、复杂布局、频繁 GC、锁等待、RecyclerView 绑定耗时、过度绘制。
- 排查应使用 Perfetto、Systrace、FrameMetrics、Choreographer 监控和业务埋点。
- 修复要基于 trace 定位具体耗时段，而不是凭经验盲改。

深挖追问：

- Choreographer 在渲染中做什么？
- Layout Inspector 能发现什么问题？
- 如何监控线上卡顿？

常见误区：

- 只用 FPS 判断体验。
- 不区分 UI 线程卡顿和 RenderThread/GPU 瓶颈。

## Q: Bitmap 内存优化有哪些关键点？

标签：高频 / 内存 / 图像

复习状态：未掌握

考察点：

- 图片解码。
- 缓存。
- 大图加载。

答案要点：

- Bitmap 内存大小与宽、高、像素格式有关，例如 ARGB_8888 每像素 4 字节。
- 加载大图应按目标尺寸采样，避免原图全量解码。
- 图片缓存通常使用内存缓存加磁盘缓存，结合 LruCache 或成熟图片库。
- 列表中要处理请求取消、复用错位、占位图和生命周期。

深挖追问：

- inSampleSize 如何计算？
- RGB_565 的优缺点是什么？
- Glide 如何感知生命周期？

常见误区：

- 只压缩文件大小，不控制解码后的内存大小。
- RecyclerView 中图片请求不处理复用。

## Q: 包体积优化可以从哪些方向做？

标签：性能 / 工程化 / 场景题

复习状态：未掌握

考察点：

- 代码、资源、so。
- R8。
- 动态交付。

答案要点：

- 代码层面可启用 R8、移除无用依赖、治理重复库。
- 资源层面可压缩图片、移除未使用资源、按密度和语言裁剪。
- native 层面要关注 ABI 数量、so 体积和符号表。
- 大型项目可考虑动态特性模块、插件化或服务端下发资源，但要权衡复杂度。

深挖追问：

- shrinkResources 和 minifyEnabled 的关系是什么？
- WebP/AVIF 适合哪些资源？
- 如何分析 APK/AAB 体积构成？

常见误区：

- 只压缩图片，不分析依赖和 so。
- 为了体积过度牺牲启动和运行性能。
